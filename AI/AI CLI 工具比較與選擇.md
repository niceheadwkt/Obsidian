在 2026 年，AI CLI（命令行介面）工具已成為開發者的標準配備，特別是對於偏好在終端機（Terminal）工作的工程師。目前市面上主流的 AI CLI 工具各具特色，從 Google 的原生整合到開源社群的靈活方案應有盡有。

以下是目前幾種主流的 AI CLI 及其核心功能比較：

### 1. **Gemini CLI (Google)**

這是目前市場上最受歡迎且「最慷慨」的工具。

- **核心功能：**
    
    - **100 萬 Token 脈絡：** 能夠處理極大型的專案代碼庫。
        
    - **Google 搜尋整合：** 可以直接連網查詢最新的文件或技術資訊。
        
    - **MCP 伺服器支援：** 透過 Model Context Protocol 擴充功能，能與 GitHub、資料庫等外部工具連動。
        
    - **Shell 模式：** 輸入 `!` 即可切換到 Shell 模式，直接執行系統指令。
        
- **特點：** 每天提供高達 1,000 次的免費請求額度，適合不想支付 API 費用的開發者。
    

### 2. **Claude Code (Anthropic)**

由 Anthropic 開發，強調極佳的使用者體驗（UX）與任務導向的自動化。

- **核心功能：**
    
    - **Agent 自動化：** 不只能寫代碼，還能自動執行測試、修復 Bug 並完成 Git Commit。
        
    - **深度專案理解：** 擅長理解複雜的架構並給予重構建議。
        
    - **終端機原生體驗：** 針對喜歡 Vim/Emacs 的工程師優化，完全無需離開命令行環境。
        
- **特點：** 模型推論能力極強（特別是程式邏輯），適合處理高難度的軟體開發任務。
    

### 3. **Codex CLI (OpenAI)**

OpenAI 針對終端機推出的原生工具，目前與 ChatGPT 的訂閱方案掛鉤。

- **核心功能：**
    
    - **雲端沙盒（Sandbox）：** 在隔離的雲端環境中運行生成的代碼，保護本地系統安全。
        
    - **語音輸入：** 支援直接用語音下達編程指令。
        
    - **模型切換：** 支援最新的 GPT-5.3 系列模型，在生成精確度上處於領先地位。
        
- **特點：** 安全性高，適合需要執行不確定腳本的情境。
    

### 4. **Aider (開源社群)**

老牌且強大的開源 AI 配對編程工具，支援多種 LLM 供應商。

- **核心功能：**
    
    - **Git 自動提交：** 每次修改後會自動生成有意義的 Commit 訊息並提交，方便回溯。
        
    - **多模型支援：** 可連接 GPT-4o、Claude 3.5、甚至本地運行的 Ollama 模型。
        
    - **雙向同步：** AI 修改代碼的同時，你也可以在 IDE 編輯，兩者不會衝突。
        
- **特點：** 靈活性最高，適合想要自定義模型來源或完全在本地端（Local）運行的用戶。

以下是針對 GitHub Copilot CLI 的安裝、設定及產製文件的詳細指南：

### 一、 安裝 GitHub Copilot CLI

目前主要透過主流的套件管理器進行安裝。請根據您的作業系統選擇指令：

- **Windows (PowerShell):**
    
    PowerShell
    
    ```
    winget install GitHub.Copilot
    ```
    
- **macOS / Linux (Homebrew):**
    

Bash

````
    brew install copilot-cli
    ```
*   **跨平台 (npm):**
    ```bash
    npm install -g @github/copilot
    ```

**驗證安裝：**
安裝完成後，輸入 `copilot version` 確認是否成功。

---

### 二、 身份驗證與登入

在使用前，您必須連結您的 GitHub 帳號（需有 Copilot 訂閱）：

1.  在終端機輸入：
    ```bash
    copilot login
    ```
2.  系統會顯示一個 **8 位數驗證碼** 並自動開啟瀏覽器（或請手動開啟連結）。
3.  輸入驗證碼並授權後，終端機將顯示 `Authenticated successfully`。

---

### 三、 專案初始化與設定（重要）

為了讓 AI 精確地產製系統文件，您需要先「初始化」專案，這會建立一個**內容指示文件**，作為 AI 理解專案的導航。

1.  **進入專案根目錄：**
    
```bash
    cd /your/project/path
    ```
2.  **執行初始化指令：**
    ```bash
    copilot init
    ```
    這會在專案中建立或更新 `.github/copilot-instructions.md`。
3.  **設定指示文件：**
    編輯此文件，加入關於專案的關鍵資訊，例如：
    *   **Architecture:** 說明系統架構（如：MVC, Microservices）。
    *   **Tech Stack:** 使用的框架與資料庫。
    *   **Coding Standards:** 命名規範或特定的文件格式要求（如：Markdown）。

> [!TIP]
> **信任目錄：** 初次執行時，Copilot 會詢問是否「信任此資料夾」，請選擇 `Yes, and remember this folder`，這能讓 AI 擁有讀取現有代碼的權限。

---

### 四、 如何產製系統文件（實戰操作）

您可以透過 **互動模式** 或 **單次指令模式** 來從現有程式碼產製文件。

#### 1. 使用互動模式（適合精細調整）
直接輸入 `copilot` 進入交談介面：
*   **動作：** 呼叫 AI 分析專案。
*   **範例指令：**
    > 「請分析目前目錄下的所有代碼，並產製一份 `SYSTEM_ARCHITECTURE.md`，內容需包含模組關係圖與 API 說明。」
*   **功能：** Copilot 會掃描程式碼，並在畫面上生成預覽。確認無誤後，您可以要求它：「將此內容儲存為文件」。

#### 2. 使用單次指令（適合快速產製）
直接從終端機下達工作：
```bash
copilot -p "根據現有程式碼生成完整的 README.md，包含安裝步驟與專案結構說明"
````

#### 3. 產製特定 API 文件 (使用 @ 符號)

如果您只需要針對某個模組產製文件，可以在指令中使用 `@` 標註檔案：

Bash

```
copilot -p "分析 @src/api/user_service.py 並產製其 API 規格文件"
```

### 五、 進階技巧：使用專門 Agent

在 2026 年版本中，Copilot CLI 支援多種內建 Agent。若要產製高品質文件，建議切換至專門的 Agent：

- **/agent doc-gen (若已安裝相關插件)：** 專門用於掃描 Docstrings 並生成網頁文件。

### 這是什麼意思？我該如何選擇？

**MCP (Model Context Protocol)** 簡單來說是讓 AI 擁有「手」和「眼」去觸碰您的開發環境。針對您目前的 Java/JSP 專案，這裡有兩個主要選項：

- **自動部署：** 當 AI 幫您寫完代碼後，它可以直接觸發 Tomcat 重新編譯或部署。
    
- **讀取日誌：** 當系統噴出 Stack Trace 時，AI 可以直接讀取 `catalina.out` 並分析錯誤原因。
    
- **環境管理：** 透過指令檢查伺服器狀態。
    

#### 2. Playwright for UI E2E

如果您有前端自動化測試需求：

- AI 可以自動撰寫並執行瀏覽器測試腳本，確認您的 JSP 頁面渲染是否正常。

## <mark style="background: #FF5582A6;">無法回答「Yes, Tomcat」</mark>

這通常是因為 CLI 介面正在等待特定的**鍵盤輸入**，而不是讓您輸入整串文字。

如果您在 PowerShell 畫面中卡住了，無法輸入 `Yes, Tomcat`，請嘗試以下幾種常見的 CLI 交互方式：

### 1. 嘗試「單字母」或「方向鍵」

- **鍵盤輸入 `y`：** 大部分的 CLI 工具只需輸入 `y` 然後按 **Enter**。
    
- **方向鍵選擇：** 觀察畫面上是否有一個小箭頭或反白區塊。如果有，請使用 **上下方向鍵** 移動到 `Tomcat/TomEE` 選項，然後按 **Space (空白鍵)** 勾選，最後按 **Enter**。
    

### 2. 如果終端機真的「凍結」了

如果按什麼都沒反應，可能是該程序暫時失去了回應。您可以：

- 按下 **`Ctrl + C`** 強制結束目前的程序。
    
- **手動跳過：** 如果您現在只想產製文件，不一定要馬上設定 MCP。您可以重新執行指令，但在後面加上跳過參數（如果該工具支援），或者在下次詢問時直接按 **Enter**（通常預設是 No）。
    

### 3. 跳過 MCP 直接產製文件

如果您現在的重點是產製文件，而不是讓 AI 幫您部署 Tomcat，其實**不需要**設定 MCP 也可以繼續。

請嘗試按下 `Ctrl + C` 退出後，直接在 PowerShell 輸入產製指令：

PowerShell

```
# 既然 AI 已經幫您建立了 .github\copilot-instructions.md
# 您可以直接針對現有的代碼下令
copilot -p "分析專案架構並產製一份 SYSTEM_MANUAL.md，請特別說明 mtStructs.xml 如何對應 JSP"
```

### 為什麼會卡住？

這類 AI CLI 工具（如圖中的輸出樣式）有時在 Windows PowerShell 下會因為**字碼集（UTF-8）**或**選取模式**導致輸入無效。

> [!TIP] **小技巧：** 如果您的 PowerShell 標題列出現「選取」字樣，代表您滑鼠點到了畫面進入了編輯模式，這時所有程式會暫停。請在畫面中按一下 **滑鼠右鍵** 或按 **Esc**，程式就會恢復執行。

3. **效能評估**：顯示快取（Cached）比例，代表下次你再問相關問題時，速度會更快。

## <mark style="background: #FF5582A6;">OpenAI  Codex CLI 與 GitHub Copilot CLI 是同樣東西?</mark>
這是兩個**不同**但有強烈血緣關係的產品。簡單來說，OpenAI Codex 是「引擎」，而 GitHub Copilot 是搭載這顆引擎並經過改裝的「跑車」。

在 2026 年的現在，它們的定位與功能差異如下：

### 1. 本質上的不同

- **OpenAI Codex (引擎/平台)：** 這是 OpenAI 開發的底層模型。最初 GitHub Copilot 就是用它來驅動的。現在它已進化成一個獨立的 **自主代理人 (Autonomous Agent)**，可以在沙盒環境中自動執行任務（例如：自動寫完一整個功能並執行測試）。
    
- **GitHub Copilot CLI (產品/工具)：** 這是 GitHub (微軟) 基於 OpenAI 模型（及其他模型）針對開發者工作流程所開發的**應用工具**。它深度整合了 GitHub 生態系（如：Repo 權限、PR 說明、Actions 等）。
    

### 2. 核心功能比較表 (2026 年版)

|**特性**|**GitHub Copilot CLI**|**OpenAI Codex CLI**|
|---|---|---|
|**角色定位**|你的「副駕駛」，輔助操作終端機。|你的「虛擬工程師」，自主完成任務。|
|**執行環境**|直接在你的電腦 (Local) 執行。|支援 **雲端沙盒 (Sandbox)**，安全執行程式。|
|**自動化程度**|較低，通常是生成指令讓你確認。|**極高**，能自動修 Bug、跑測試、直到成功。|
|**瀏覽器控制**|無。|**強項**，可透過外掛直接操作 Chrome。|
|**整合度**|與 GitHub 儲存庫、PR 完美整合。|與 OpenAI API、ChatGPT 訂閱連結。|
|**費用**|包含在 Copilot 訂閱中 (有免費額度)。|消耗 ChatGPT Plus 額度或 API Tokens。|

### 3. 我該選哪一個？

- **選 GitHub Copilot CLI 的時機：**
    
    - 你主要在本地進行開發。
        
    - 你需要 AI 幫你解釋複雜的 Shell 指令或 Git 操作。
        
    - 你已經有 GitHub Copilot 訂閱，想要在終端機也能用到它的功能。
        
- **選 OpenAI Codex CLI 的時機：**
    
    - 你需要處理**多檔案、跨模組**的大型重構任務。
        
    - 你希望 AI 在**安全沙盒**中先跑過一遍程式碼，確定沒錯再交給你。
        
    - 你需要 AI 協助網頁自動化或跨應用程式的工作（如：從 Salesforce 抓資料到程式裡）。
        

### 總結

雖然 GitHub Copilot 最初是基於 Codex 技術，但到 2026 年，**GitHub Copilot 偏向「終端機助手」**，而 **OpenAI Codex 已發展為「全自動開發代理人」**。兩者在功能上已經有明顯的市場區隔。

## <mark style="background: #FF5582A6;">Get-ChildItem -Path "C:\aiTest\mt" -Recurse -Directory | Select-Object FullName | Sort-Object FullName | Format-Table -AutoSize 這何意?</mark>
這行指令是 PowerShell 的「專案地圖掃描」**指令。簡單來說，它的目的是**列出您專案中所有的資料夾路徑，並整理成一張整潔的清單，好讓 AI 或開發者能快速掌握整個系統的目錄結構。

以下是這條指令的詳細拆解：

### 1. 指令分解說明

- **`Get-ChildItem -Path "C:\aiTest\mt"`**
    
    - 這是在說：「去 `C:\aiTest\mt` 這個資料夾裡面找東西。」
        
- **`-Recurse`**
    
    - **「遞迴掃描」**。不只看第一層，連子資料夾、子資料夾的子資料夾...全部都挖出來。
        
- **`-Directory`**
    
    - **「只找資料夾」**。過濾掉所有的檔案（如 `.java` 或 `.xml`），只看目錄結構。
        
- **`Select-Object FullName`**
    
    - 只要看「完整路徑」。這會過濾掉修改日期、大小等雜訊，只留下路徑文字。
        
- **`Sort-Object FullName`**
    
    - 按字母「由 A 到 Z 排序」，讓結構看起來更直觀。
        
- **`Format-Table -AutoSize`**
    
    - **「格式化為表格」**，並自動調整寬度，確保路徑不會被截斷，方便閱讀。
        

### 2. 為什麼 AI CLI 需要執行這行指令？

當您要求 AI 產製系統文件時，它必須先知道您的**專案架構（Project Architecture）**。透過這行指令，AI 可以立刻得知：

- **套件結構：** 像是 `com/icsc/mt/dao` 或 `com/icsc/mt/vo` 在哪裡。
    
- **配置位置：** `config` 或 `xml` 資料夾是否存在。
    
- **備份機制：** 是否有 `backup` 資料夾。
    

這就是為什麼在您之前的截圖中，AI 能夠列出「10 大業務領域」或「DAO/VO 模式」的大綱，因為它透過這行指令看穿了您專案的骨架。

### 3. 您可以如何利用它？

如果您想自己確認目前的目錄結構，直接把這行複製貼上到 PowerShell 執行，您會得到類似這樣的輸出：

|**FullName**|
|---|
|C:\aiTest\mt\config|
|C:\aiTest\mt\src\com\icsc\mt\dao|
|C:\aiTest\mt\src\com\icsc\mt\vo|
|C:\aiTest\mt\web\jsp|

### 💡 實戰小建議

如果您在執行這行指令時又遇到 **"Permission denied"**，這通常是因為 `-Recurse` 掃描到了某些系統隱藏資料夾。您可以嘗試在指令最後加上 `-ErrorAction SilentlyContinue` 來跳過報錯：

PowerShell

```
Get-ChildItem -Path "C:\aiTest\mt" -Recurse -Directory -ErrorAction SilentlyContinue | Select-Object FullName | Sort-Object FullName | Format-Table -AutoSize
```

這能確保指令順利跑完，讓 AI 拿到它要的「地圖」！

## <mark style="background: #FF5582A6;">Claude Code我如何中斷</mark>

在中斷或結束 **Claude Code** 時，根據您目前是在「輸入中」、「AI 思考中」還是「想直接關機」，有幾種不同的方式。

針對您在 Windows PowerShell 的環境，請參考以下操作：

### 1. 停止目前的「思考」或「生成」（不關閉程式）

如果您發現 AI 寫錯了，或產製的文件太長您想喊卡：

- **快捷鍵：`Ctrl + C`**
    
    - 這是標準的中斷指令。按一下通常會停止當前的文字生成或工具執行（如掃描檔案），並讓您回到可輸入的對話框。
        
    - **注意：** 如果 AI 正在執行複雜的系統指令，有時需要連按兩次。
        

### 2. 結束並退出 Claude Code（關閉程式）

當您完成工作，想回到一般的 PowerShell 提示字元：

- **輸入指令：`/exit`** 或 **`/quit`**
    
    - 這是最優雅的結束方式，會確保所有暫存檔案與快取都正確處理。
        
- **快捷鍵：`Ctrl + D`**
    
    - 這是終端機通用的 EOF（結束符號），效果等同於輸入 `/exit`。
        

### 3. 強制中斷背景任務 (Subagents)

在 2026 年版本中，Claude 有時會啟動多個「小助手（Subagents）」在背景幫您寫程式：

- **快捷鍵：`Ctrl + X` 接著按 `Ctrl + K`**
    
    - 這會立即殺死所有正在背景運行的子任務。
        

### 4. 如果 Claude Code 完全「凍結」沒反應

如果您按了 `Ctrl + C` 還是沒反應（這在 Windows PowerShell 偶爾會發生）：

1. **解除「選取模式」：**
    
    先看視窗標題有沒有「選取」字樣。有的話，按一下鍵盤的 **`Esc`**，程式通常就會恢復。
    
2. **暴力結束（最快）：**
    
    直接**關閉 PowerShell 視窗**，或者開啟另一個新的 PowerShell 視窗輸入以下指令：
    
    PowerShell
    
    ```
    Stop-Process -Name "claude" -Force
    ```
