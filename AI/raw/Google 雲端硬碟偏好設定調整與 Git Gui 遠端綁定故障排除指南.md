
本指南記錄了在公司筆記型電腦（NB）上，針對 Google 雲端硬碟電腦版同步模式的檢視，以及在使用 Git Gui 進行 GitHub 遠端儲存庫綁定時發生的連線錯誤之完整排解流程。

## 一、 Google 雲端硬碟電腦版偏好設定檢視

### 1. 目前設定狀態確認
根據偏好設定截圖（`2026-06-23 15 59 25.jpg`），目前公司筆記型電腦的「我的雲端硬碟同步處理選項」確實勾選為**「串流檔案」**模式。

### 2. 「串流檔案」模式的運作特性
* **空間佔用少**：檔案實體儲存在雲端，本機僅顯示虛擬的投影。
* **即時下載**：只有在開啟檔案或手動設定為「可離線使用」時，檔案才會真正下載並佔用硬碟空間。
* **存取路徑**：透過電腦上的虛擬磁碟機（通常為 G 槽）或特定資料夾進行存取。

### 3. 對 Git 版本控制的潛在影響與潛在風險
「串流檔案」模式對於一般的文書處理影響不大，但**強烈不建議用於 Git 倉庫（如 Obsidian 筆記結合 Git 備份）的存放路徑**，原因如下：
* **頻繁的網路 IO 衝突**：Git 在執行狀態掃描（Rescan）、暫存（Stage）或提交（Commit）時，需要瞬間讀寫 `.git` 目錄下成百上千個微小的索引碎片檔案。
* **產生大量同步暫存檔**：雲端硬碟軟體為了處理這些頻繁的微小變動，會在背景不斷產生 `.tmp.driveupload` 等隱藏的同步暫存資料夾，這會導致 Git 偵測到大量非必要的「未追蹤變更」，進而干擾工作區的乾淨度。
* **檔案鎖定與損壞風險**：在網路延遲或雲端同步中途，Git 寫入索引若與雲端同步程序衝突，容易導致檔案被鎖定、Git 運作極度緩慢，嚴重時甚至會引發 Git 索引損壞（Corrupted index）。

### 🛠️ 行動建議：變更為「雙向同步檔案」
若欲實現流暢且穩定的 Git 備份，建議在該設定介面中進行以下調整：
1. 將選項改勾選為**「雙向同步檔案」**（即鏡像模式）。
2. 此模式會將雲端檔案 100% 實體下載至本機實體硬碟（例如 `C:\Users\ch26788\我的雲端硬碟`）。
3. 點擊右下角「儲存」並依系統提示重新啟動雲端硬碟軟體。這能讓 Git 讀寫回歸純粹的本機高速操作，徹底根除同步暫存檔的干擾。

## 二、 Git Gui 遠端倉庫綁定錯誤排除

### 1. 錯誤現象分析
根據 Git Gui 截圖（`2026-06-23 16 15 44.jpg`），在執行 `Remote -> Add...` 欲綁定 GitHub 遠端儲存庫 `https://github.com/niceheadwkt/Obsidian.git` 時，跳出如下錯誤視窗：
> **Failed to add remote 'origin' of location 'https://github.com/niceheadwkt/Obsidian.git'.**

### 2. 錯誤根本原因
在「Add New Remote」的設定視窗中，下方 **Further Action** 預設勾選了 **`Fetch Immediately`（立即抓取）**。
由於目標 GitHub 儲存庫已設為 **Private（私有）**，當 Git Gui 嘗試「立刻」連線至 GitHub 抓取資料時，因**此時本地端尚未通過 GitHub 的帳號密碼或 Token 安全驗證**，連線直接遭到 GitHub 伺服器拒絕，進而導致整個遠端登記動作被判定為失敗。

### 🛠️ 標準解決步驟

#### 步驟 A：調整遠端加入策略（跳過即時檢查）
1. 點擊錯誤視窗的「確定」將其關閉。
2. 重新點擊 Git Gui 上方選單的 **`Remote`** ──> **`Add...`**。
3. 重新輸入 Name（`origin`）與 Location（`https://github.com/niceheadwkt/Obsidian.git`）。
4. ✨ **關鍵調整**：將下方的 Further Action 改勾選為 **`Do Nothing Else Now`（現在什麼都不做）**。
5. 點擊 **`Add`**。此時 Git 僅會在本地配置設定檔（`.git/config`），跳過預先網路連線，即可順利新增成功。

#### 步驟 B：排查名稱衝突（若步驟 A 依然失敗）
如果調整後仍跳出相同錯誤，代表本地倉庫過去曾綁定過其他遠端網址，導致 `origin` 這個名字已被佔用。請依以下方式清理：
1. 點擊 Git Gui 上方選單的 **`Remote`** ──> 點選 **`Remove...`**。
2. 在彈出的視窗中檢查是否有已存在的 `origin`，將其勾選並刪除。
3. 清除乾淨後，重新執行「步驟 A」即可。

## 三、 進階核心補充：GitHub Private 倉庫的 Push 授權驗證

當遠端網址成功登記後，由於儲存庫權限為 **Private**，在 Git Gui 中首次點擊 **`Push`**（推送）時，必須通過權限校驗，否則會發生認證失敗。

### 1. 產生 GitHub 個人訪問權限 Token (PAT)
1. 登入 GitHub 網頁 ──> 點選右上角個人頭像 ──> 進入 **Settings**。
2. 於左側選單捲動至最下方，點擊 **Developer settings**。
3. 選擇 **Personal access tokens** ──> **Tokens (classic)**。
4. 點擊 **Generate new token** ──> **Generate new token (classic)**。
5. **Note** 輸入用途描述（如：`Work_NB_GitGui`），Expiration（過期時間）依需求設定。
6. **Select scopes（權限）**：務必勾選 **`repo`**（包含其下所有子項目）。
7. 拉到最下方點擊 **Generate token**，並**複製保存**畫面上生成的 `ghp_` 開頭長串金鑰（該金鑰僅顯示一次）。

### 2. 於 Git Gui 執行首次 Push
1. 回到 Git Gui，點擊正下方的 **`Push`** 按鈕。
2. 當系統彈出驗證視窗時：
   * **Username**：輸入您的 GitHub 帳號名稱（`niceheadwkt`）。
   * **Password**：**請勿輸入 GitHub 登入密碼，請直接貼上剛剛產生的 `ghp_` 開頭 Token 金鑰**。
3. 驗證成功後，Windows 憑證管理員會自動記憶此憑證，後續執行 Push 即可直接一鍵推送，無需重複輸入。
