
本指南完整收錄並精煉了在 VS Code 與 Git GUI 環境下，因多重隱藏 Git 儲存庫（Nested Repositories）所引發的本地端功能異常（如 GitLens 誤判為 `Untracked`）、最外層 `.gitignore` 的防範規則配置，以及解決 Git 推送遭拒與資料庫清理鎖定（`git gc` 失敗）的實務修復方案，並包含 GitHub 網頁端跨版本比對技術解密。

---

## 一、 核心故障：VS Code 內 Git 偵測機制混亂與誤判分析

### 1. 故障現象描述
在 VS Code 中開啟筆記檔案 `Git GUI 與 GitHub 雙向同步實務.md` 時，儘管該檔案事實上已成功提交並推送（Push）至 GitHub 遠端儲存庫達 3 次之多，但本地端的 VS Code 側邊欄 **GITLENS INSPECT** 的 **FILE HISTORY** 區塊卻發生以下異常：
* 僅顯示 `Uncommitted changes`。
* 滑鼠懸停於檔案上時，系統提示彈窗顯示為 `Working Tree ・ Untracked`（未追蹤狀態）。
* 無法順利在本地端讀取或比對過往的歷史 Commit 節點。

### 2. 根本原因剖析（多重 Git 儲存庫嵌套）
經實地盤點本機檔案路徑與目錄架構後，發現關鍵原因在於本地端存在**兩個互相嵌套的獨立 Git 儲存庫**：

| 儲存庫主體 | 實體資料夾路徑 | 預設分支 | 角色與預期功能 |
| :--- | :--- | :--- | :--- |
| **主儲存庫（正確）** | `/Obsidian/` | `main` | 負責完整知識庫、全分類子資料夾（含 AI 分類）的所有版本控制與 GitHub 同步。 |
| **子儲存庫（衝突源）** | `/Obsidian/AI/` | `master` | 因過去下載範例專案、初始設定或不慎執行 `git init` 而意外產生的獨立 Git 節點。 |

當在 VS Code 中點開位於 `/Obsidian/AI/wiki/` 下的筆記時，VS Code 機制會「向下優先識別」最接近該檔案的儲存庫設定。因此，**GitLens 優先被 `AI/` 資料夾內獨立的 `.git` 隱藏資料夾所攔截**。由於該筆記的 3 次提交紀錄都是記錄在最外層的 `Obsidian (main)` 儲存庫中，對於 `AI (master)` 子儲存庫而言，這份檔案從未被提交過，才會導致本地端判定其為完全陌生的 `Untracked` 狀態。

### 3. 伴隨發生的異常環境資料夾
在 `/Obsidian/AI/` 子資料夾下，除了衝突的 `.git` 外，還伴隨產生了以下不必要的隱藏系統資料夾，其成因與清理對策如下：
1. **`.git`（強烈建議刪除）：** 子儲存庫的核心設定。刪除後，GitLens會自動向上追溯，重新與外層正確的 `Obsidian (main)` 儲存庫進行綁定，立刻恢復正常的過往歷史紀錄檢視。
2. **`.obsidian`（可以刪除）：** Obsidian 軟體的設定檔。代表過去曾不小心將 `AI` 資料夾單獨作為獨立的「儲存庫 (Vault)」於軟體中開啟。真正的主設定檔存放於 `/Obsidian/.obsidian`，故此處的可以安全移除。
3. **`.trash`（可以刪除）：** Obsidian 專用的軟體垃圾桶暫存區。配合上述獨立庫誤開而產生，屬於無用垃圾檔案，刪除可釋放空間。

## 二、 進階防範：最外層 `.gitignore` 阻斷子目錄 Git 偵測規則

為了根治並防止未來在 `AI/` 等子目錄下下載外部開源專案、MCP 伺服器或範例工具時，再度自動生成全新的 `.git` 資料夾干擾主儲存庫，應在**最外層（`/Obsidian/` 根目錄下）的 `.gitignore` 檔案**中加入以下精準阻斷規則：

```text
# ====================================================================
# 知識庫巢狀 Git 衝突防範規則
# ====================================================================

# 強制忽略任何子資料夾底下的隱藏 .git 資料夾，防止本地端偵測機制（VS Code / GitLens）錯亂
**/.git/

# (選填) 如果未來有特定 AI 子專案完全不需要同步到主儲存庫，可直接整條排除：
# AI/projects/外部獨立專案名稱/
```

### 🛠️ 核心快取刷清命令
**重要：** 如果某些子目錄的 `.git` 殘留狀態已經被外層主儲存庫誤當成檔案追蹤（Tracked），單純修改 `.gitignore` 是不會生效的。請在 `/Obsidian/` 根目錄下開啟終端機執行以下命令，強制刷清 Git 快取：
```bash
# 1. 僅清除 Git 的追蹤快取索引（不會刪除硬碟上的實體筆記檔案）
git rm -r --cached .

# 2. 依照全新修訂的 .gitignore 規則重新捕捉檔案
git add .

# 3. 提交此項環境優化變更
git commit -m "chore: 配置防嵌套 .gitignore 規則並刷新 Git 索引快取"
```

## 三、 實務錯誤排除：Git 推送遭拒與雲端鎖定衝突解決

### 1. 錯誤（一）：推送遭拒（[rejected] main -> main fetch first）
當您使用本地端的 Git GUI 點擊 `Push` 準備將變更推送到 GitHub 時，跳出紅色區塊的 `Error: Command Failed` 視窗，提示遠端包含本地端沒有的變更。

#### 🛠️ 標準作業程序（SOP）：
1. **先在 Git GUI 內提交本機修改：** 將左側所有「Unstaged Changes（橘色）」移至「Staged Changes（綠色）」，輸入 Commit Message 並點擊 **Commit**，讓工作區變乾淨。
2. **終端機一鍵變基合併：** 在 `/Obsidian/` 根目錄開啟終端機，執行以下指令：
   ```bash
   git pull --rebase origin main
   ```
3. **回 Git GUI 順暢 Push：** 合併成功後，回到 Git GUI 點擊 **Push** 即可上傳成功。

### 2. 錯誤（二）：清理資料庫失敗（Deletion of directory .../.git/objects/00 failed）
在 Git GUI 中執行壓縮或清理資料庫（`git gc`）時，跳出 Question 視窗提示刪除暫存資料夾失敗。

#### 根本原因：雲端硬碟鎖定衝突
您的專案位於 **Google 雲端硬碟 (Google Drive)** 內。當 Git 本地端底層頻繁建立或刪除暫存資料庫碎片（Object block）時，雲端同步軟體一偵測到變動便會**立刻強制鎖定該檔案**準備上傳。此時 Git GUI 想要回頭刪除或移動該內部碎片，就會被 Windows 系統拒絕而卡死。

#### 🛠️ 排障對策：
1. 點擊 Windows 右下角工作列的 **Google 雲端硬碟（小雲朵圖示）**。
2. 點擊右上角 **「設定（齒輪圖示）」** -> 點選 **「暫停同步 (Pause Syncing)」**。
3. 回到 Git GUI 卡住的提示視窗，點擊 **「Yes」** 讓它重試（此時雲端軟體已釋放鎖定，能秒速清理完成）。
4. 清理與 Push 結束後，回到雲端硬碟設定點選 **「恢復同步 (Resume Syncing)」**。

## 四、 GitHub 網頁端跨版本比對實務技術

當本地端正在調整環境時，GitHub 網頁端提供了極度穩定的跨版本對比功能。若要針對某一檔案的**第 1 版（最初版，6/13）**與**第 3 版（最新版，6/23）**進行深度比對，有以下兩種高效操作模式：

### 1. 終極快速：手動拼湊網址對比法（SHA 雜湊值應用）
GitHub 的版本對比核心邏輯為在網址後方帶入：`compare/舊版本SHA...新版本SHA`。針對本專案之特定版本雜湊值（6碼短 SHA 碼）：
* **第 1 版（舊版本 SHA）：** `055c45b`
* **第 3 版（新版本 SHA）：** `65d3fb5`

直接在瀏覽器網址列貼上並執行下方精準網址，即可直達比對頁面：
```text
https://github.com/niceheadwkt/Obsidian/compare/055c45b...65d3fb5
```

### 2. 免打字：全滑鼠點擊圖形化比對法
1. 點擊 GitHub 左上角專案名稱 **Obsidian** 回到儲存庫根目錄首頁。
2. 依序點選進入檔案路徑：`AI` -> `wiki` -> `Git GUI 與 GitHub 雙向同步實務.md`。
3. 點擊該檔案視窗右上角的 **History**（歷史紀錄）按鈕，此時會列出過往三次的 Commit 紀錄清單。
4. 找到第 1 版紀錄，點擊其右側的**兩張紙重疊圖示（Copy full SHA）**，將其完整雜湊值複製到剪貼簿。
5. 將網頁向上捲動，點擊畫面上方的 **Pull requests** 標籤頁面，並點擊綠色的 **New pull request** 按鈕。
6. 在比對控制列中，點擊右側的 **compare: main** 下拉選單，在搜尋方塊中直接**按右鍵貼上**剛剛複製的第 1 版 SHA 碼並按 Enter 回車。
7. GitHub 系統將會自動將基礎版本（Base）設為第 1 版，與當前最新版進行完整的差異圖形化渲染。

## 五、 GitHub 差異檢視介面進階解讀

### 1. Split View（左右分欄）模式切換
GitHub 預設採用 Unified（上下統整）模式。要切換為更適合長篇 Markdown 筆記閱讀的左右分欄模式：
* **操作路徑：** 於差異檢視畫面正上方，找到 `Top` 與 `Search within code` 右側的 **齒輪圖示 (Settings)** -> 點選下拉選單中 **Diff view** 下的 **Split** 項目。畫面將切換為左半邊顯示舊版（紅色刪除）、右半邊顯示新版（綠色新增）。

### 2. Hunk Header（程式碼區塊標頭）深度解析
在差異檢視中，畫面出現的藍色橫條長框在 Git 機制中稱為 **Hunk Header**。其核心用途是提示使用者「目前顯示的片段在原文章中的行數位置」，並自動隱藏中間未經修改的冗長段落。

以藍色橫條框中出現的典型代碼為例進行拆解：
```text
@@ -4,10 +4,11 @@ tags: [Git, GitGUI, GitHub, 版本控制, 同步, 衝突解決]
```
* **`@@ -4,10 @@`（左半邊舊版指標）：** `-` 代表舊版本。意即目前下方顯示的舊文字片段，是從舊版筆記的**第 4 行**開始，總共連續向下顯示了 **10 行** 的內容。
* **`@@ +4,11 @@`（右半邊新版指標）：** `+` 代表新版本。意即目前右側顯示的新文字片段，是從新版筆記的 **第 4 行** 開始，總共連續向下顯示了 **11 行**（行數增加代表此區塊內有新增文字）。
* **淡淡的背景文字（如 `tags: [...]`）：** Git 核心演算法自動抓取此區塊最鄰近的 Markdown 標題或 Front Matter 首行文字，做為視覺上的導覽標籤，方便使用者辨識該修改位於文章的哪一個章節段落。
* **最左側的「上下雙向小箭頭」：** 點擊橫條左側的向上或向下小箭頭，可以將中間被 GitHub 自動隱藏的、未修改的前後文筆記內容逐步展開，無須切換回完整檔案即可通觀上下文。
