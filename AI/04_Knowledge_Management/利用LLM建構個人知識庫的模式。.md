# LLM Wiki

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#llm-wiki)

利用LLM建構個人知識庫的模式。

這是一個概念文件，旨在複製並貼上到您自己的LLM代理（例如OpenAI Codex、Claude Code、OpenCode/Pi等）中。它的目標是傳達高層次的概念，但您的代理商將與您協作建立具體細節。

## 核心思想

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#the-core-idea)

大多數人使用學習型學習模型（LLM）和文件的經驗都類似於隨機產生（[[RAG]]）：你上傳一系列文件，LLM 在查詢時檢索相關片段，並產生答案。這種方式雖然可行，但 LLM 每次都要從頭開始重新發現知識，沒有累積。如果你提出一個需要綜合五個文件的複雜問題，LLM 每次都必須找到並拼湊相關的片段，沒有任何知識累積。 NotebookLM、ChatGPT 檔案上傳以及大多數 RAG 系統都是這樣運作的。

這裡的理念有所不同。 LLM 並非僅僅在查詢時從原始文件中檢索訊息，而是**逐步建立並維護一個持久的維基**——一個結構化的、相互連結的 Markdown 文件集合，它位於用戶和原始資料來源之間。當您添加新的資料來源時，LLM 不僅會對其進行索引以便後續檢索，還會讀取它，提取關鍵信息，並將其整合到現有的維基中——更新實體頁面，修訂主題摘要，指出新數據與舊說法相矛盾之處，從而強化或挑戰不斷演進的綜合分析。知識只需編譯一次即可_保持最新_，無需在每次查詢時重新推導。

這就是關鍵區別：**維基百科是一個持續更​​新、不斷改進的資源。**交叉引用已經存在，矛盾之處也已被標記，綜合資訊已經反映了你閱讀過的所有內容。你新增的每一個來源、提出的每一個問題都會讓維基百科的內容更加豐富。

你幾乎從不（或很少）親自編寫維基——所有內容都由LLM（知識庫管理員）編寫和維護。你負責尋找資料、探索和提出正確的問題。 LLM負責所有繁瑣的工作——總結、交叉引用、歸檔和記賬，這些工作使知識庫能夠長期發揮作用。實際上，我一邊打開LLM代理，一邊打開Obsidian。 LLM根據我們的對話進行編輯，我即時瀏覽結果——點擊連結、查看圖表視圖、閱讀更新後的頁面。 Obsidian是整合開發環境（IDE）；LLM是程式設計師；維基就是程式碼庫。

這可以應用在許多不同的情境。以下是一些例子：

- **個人**：追蹤自己的目標、健康、心理、自我提升——記錄日記、文章、播客筆記，並隨著時間的推移建構一個結構化的自我形象。
- **研究**：對某個主題進行數週或數月的深入研究——閱讀論文、文章、報告，並逐步建立一個包含不斷發展的論點的綜合維基。
- **閱讀一本書**：邊讀邊整理章節，建立頁面記錄人物、主題、情節線索以及它們之間的連結。最終，你將擁有一個內容豐富的配套維基。想想像[托爾金門戶網站（Tolkien Gateway）](https://tolkiengateway.net/wiki/Main_Page)這樣的粉絲維基——成千上萬個相互關聯的頁面，涵蓋人物、地點、事件、語言等等，由志願者社區歷經數年構建而成。你可以在閱讀的同時，親手建構類似的內容，而LLM（圖書館學碩士）則負責所有的交叉引用和維護工作。
- **業務/團隊**：一個由LLM維護的內部wiki，內容來自Slack討論串、會議記錄、專案文件和客戶電話。可能也會有人工參與審核更新。 wiki之所以能保持最新，是因為LLM負責團隊中其他人都不願意做的維護工作。
- **競爭分析、盡職調查、旅行計劃、課程筆記、興趣愛好深度研究**——任何需要你隨著時間的推移積累知識，並且希望將其整理成冊而不是散亂無章的事物。

## 建築學

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#architecture)

它分為三層：

**原始資源**－您精心收集的來源文件。包括文章、論文、圖像和資料檔。這些資源是不可更改的——LLM 系統會讀取它們，但絕不會對其進行修改。這就是您的真實來源。

**維基**－一個由LLM產生的Markdown檔案目錄。內容包括摘要、實體頁面、概念頁面、比較、概述和綜合。 LLM完全掌控這一層。它創建頁面，在新資源到來時更新頁面，維護交叉引用，並確保所有內容的一致性。您閱讀它，LLM編寫它。

**模式**檔案（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md）告訴 LLM wiki 的結構、約定俗成的規則以及在匯入資源、回答問題或維護 wiki 時應遵循的工作流程。這是關鍵的設定檔——它使 LLM 成為一個規範的 wiki 維護者，而不是一個普通的聊天機器人。隨著時間的推移，您和 LLM 會共同完善這個模式文件，並不斷探索適合您領域的有效方法。

## 營運

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#operations)

**導入。**您將新來源新增至原始集合中，並指示 LLM 進行處理。流程範例：LLM 讀取來源，與您討論關鍵要點，在 wiki 中編寫摘要頁面，更新索引，更新 wiki 中相關的實體和概念頁面，並將條目新增至日誌中。單一來源可能會影響 10-15 個 wiki 頁面。我個人更喜歡一次導入一個來源並全程參與——我會閱讀摘要，檢查更新，並指導 LLM 重點關注哪些內容。當然，您也可以一次批次匯入多個來源，這樣可以減少監督。您可以根據自己的習慣開發工作流程，並將其記錄在模式中，以便將來使用。

**查詢。**您在維基中提出問題。 LLM 會搜尋相關頁面，讀取它們，並綜合整理出包含引用的答案。答案可以根據問題的不同而呈現不同的形式——Markdown 頁面、對比表格、幻燈片（Marp）、圖表（matplotlib）或畫布。重要的一點是：**優秀的答案可以作為新頁面歸檔到維基。**您提出的對比、分析、發現的關聯性——這些都很有價值，不應該消失在聊天記錄中。這樣，您的探索就像已匯入的資源一樣，不斷累積在知識庫中。

**定期檢查。**請LLM定期對wiki進行健康檢查。檢查內容包括：頁面之間的矛盾、已被新資料取代的過時說法、沒有外部連結的孤立頁面、提及但缺少獨立頁面的重要概念、缺失的交叉引用以及可以透過網路搜尋填補的資料空白。 LLM擅長提出新的研究問題和新的參考資料。這有助於wiki在發展過程中保持健康。

## 索引和日誌記錄

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#indexing-and-logging)

隨著維基百科的不斷擴充，有兩個特殊文件可以幫助LLM（以及您）更好地瀏覽維基百科。它們各有不同的用途：

**index.md**是一個面向內容的索引。它是一個維基百科所有內容的目錄——每個頁面都包含一個連結、一行摘要，以及可選的元數據，例如日期或來源數量。索引按類別（實體、概念、來源等）組織。 LLM 會在每次資料攝取時更新它。當回應查詢時，LLM 首先會讀取索引以查找相關頁面，然後深入查看這些頁面。這種方法在中等規模（約 100 個來源，約數百個頁面）下效果出奇地好，並且避免了使用基於嵌入的 RAG 架構。

**log.md 檔案**依時間順序排列。它只追加記錄事件及其發生的時間—例如內容匯入、查詢和程式碼檢查。一個實用技巧：如果每個條目都以一致的前綴開頭（例如 `/` `## [2026-04-02] ingest | Article Title`），則可以使用簡單的 Unix 工具解析該日誌——例如，`log.md -5` 即可`grep "^## \[" log.md | tail -5`取得最近的 5 個條目。此日誌提供了 wiki 的發展歷程，並有助於 LLM 了解最近的工作內容。

## 可選：命令列工具

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#optional-cli-tools)

在某些時候，您可能需要建立一些小型工具來幫助 LLM 更有效率地操作 wiki。最顯而易見的就是 wiki 頁面搜尋引擎——在小規模時，索引檔案就足夠了，但隨著 wiki 的成長，您需要更完善的搜尋功能。 qmd[是](https://github.com/tobi/qmd)一個不錯的選擇：它是一個本地 Markdown 檔案搜尋引擎，支援混合式 BM25/vector 搜尋和 LLM 重新排名，所有操作都在裝置端完成。它既有命令列介面 (CLI)（LLM 可以透過 shell 呼叫它），也有 MCP 伺服器（LLM 可以將其作為原生工具使用）。您也可以自己建立一個更簡單的工具——LLM 可以在需要時幫助您編寫一個簡單的搜尋腳本。

## 技巧和竅門

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#tips-and-tricks)

- **[[Obsidian]] Web Clipper**是一款瀏覽器擴充程序，可將網頁文章轉換為 Markdown 格式。對於快速將資源添加到您的原始收藏中非常有用。
- **將圖片下載到本機。**在 Obsidian 設定 → 檔案和連結中，將「附件資料夾路徑」設定為固定目錄（例如 `/etc/ `raw/assets/`mds/`）。然後在設定 → 熱鍵中，搜尋“下載”，找到“下載目前檔案的附件”並將其綁定到一個熱鍵（例如 Ctrl+Shift+D）。剪輯文章後，按下熱鍵，所有圖片都會下載到本機磁碟。雖然此步驟是可選的，但非常實用——它允許 LLM 直接查看和引用圖片，而無需依賴可能失效的 URL。請注意，LLM 無法一次讀取包含內聯圖片的 Markdown 文本——解決方法是讓 LLM 先讀取文本，然後單獨查看部分或全部引用的圖片以獲取更多上下文信息。這種方法略顯繁瑣，但效果尚可。
- **Obsidian 的圖形視圖**是查看 wiki 結構的最佳方式——哪些內容相互連接，哪些頁面是中心頁面，哪些頁面是孤立頁面。
- **Marp**是一種基於 Markdown 的幻燈片格式。 Obsidian 有對應的插件。它可用於直接從 wiki 內容產生簡報。
- **Dataview**是一個 Obsidian 插件，用於對頁面 frontmatter 進行查詢。如果您的 LLM 系統為 wiki 頁面新增了 YAML frontmatter（標籤、日期、來源計數），Dataview 可以產生動態表格和清單。
- 這張維基百科其實就是存放 Markdown 文件的 Git 倉庫。您可以免費獲得版本歷史記錄、分支管理和協作功能。

## 為什麼這種方法有效

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#why-this-works)

維護知識庫最繁瑣的部分並非閱讀或思考，而是記錄。更新交叉引用、保持摘要的時效性、記錄新數據與舊說法的矛盾之處、確保數十個頁面內容的一致性，這些都至關重要。人們放棄維基百科，是因為維護負擔的成長速度超過了其價值的成長速度。而LLM（知識庫管理員）不會感到厭倦，不會忘記更新交叉引用，而且一次就能處理15個檔案。維基百科之所以能夠持續維護，是因為其維護成本幾乎為零。

人的職責是篩選資料、指導分析、提出好問題，並思考這一切的意義。法學碩士的職責則是其他一切。

這個想法在精神上與範內瓦·布希的Memex（1945）一脈相承——Memex是一個個人化的、精心維護的知識庫，文件之間存在關聯。布希的設想比後來的網路更接近於此：私密的、主動維護的，文件之間的關聯與文件本身同樣重要。他未能解決的問題是誰來維護。 LLM專案則負責解決這個問題。

## 筆記

[](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f#note)

本文檔有意保持抽象。它描述的是概念，而非具體的實現方式。具體的目錄結構、模式約定、頁面格式、工具等等，都將取決於您的領域、您的偏好以及您選擇的LLM（生命週期管理）系統。以上所有內容都是可選且模組化的——選擇有用的部分，忽略不需要的部分。例如：您的資源可能只有文本，因此您完全不需要圖像處理功能。您的wiki可能很小，只需要索引文件，無需搜尋引擎。您可能不需要投影片，只需要Markdown頁面。您可能需要完全不同的輸出格式。使用本文檔的正確方法是與您的LLM代理程式共用，並共同協作，建立符合您需求的版本。本文檔的唯一功能是傳達模式。您的LLM可以處理其餘部分。

### 一、 主題（Theme）

**「建立基於 LLM 的個人/團隊持續複利知識庫架構」** 探討如何利用 LLM（大型語言模型）Agent 來自動建立、維護並更新一個結構化、具備雙向連結的 Markdown 知識庫（如 Obsidian），從而克服傳統 RAG（檢索增強生成）無法累積知識與融會貫通的缺點。

### 二、 核心重點（Key Takeaways）

#### 1. 傳統 RAG 的痛點 vs. LLM-Wiki 的創新

- **傳統 RAG / 文件上傳：** 屬於「即時檢索」。每次問問題時，AI 才臨時從碎裂的文件片段中找答案，並未真正「內化」知識。如果問題需要綜合 5 份文件的內容，AI 每次都要重新拼湊，知識無法累積。
    
- **LLM-Wiki 的概念：** 屬於「事前編譯、持續演進」。當放入新資料時，LLM 會主動閱讀、提取重點、更新舊頁面、抓出新舊資訊的矛盾、並建立雙向連結。**知識只會越累積越深厚，而不是每次都從頭來過。**
    

#### 2. 三層架構（Architecture）

- **原始資料層（Raw Sources）：** 論文、文章、會議記錄。這部分是「唯讀」的，AI 絕不修改，作為事實的單一來源（Source of Truth）。
    
- **Wiki 知識層（The Wiki）：** 由 LLM 全權負責編寫與維護的 Markdown 資料夾。包含主題總結、實體頁面、概念比對等。
    
- **規範層（The Schema）：** 專門寫給 LLM Agent 遵守的規則文件（如 `CLAUDE.md`）。裡面定義了知識庫的結構、命名規範、以及 AI 處理資料的標準流程（Workflow）。
    

#### 3. 三大核心運作模式（Operations）

- **資料匯入（Ingest）：** 每丟入一個新來源，LLM 就會去更新可能相關的 10~15 個 Wiki 頁面，自動打散並融入現有體系。
    
- **查詢內化（Query）：** 使用者向 Wiki 發問，LLM 給出答案後，**有價值的分析與解答會被重新存回 Wiki 變成新的頁面**，讓對話歷史不會消失，轉為永久知識。
    
- **定期檢修（Lint）：** 定期叫 LLM 檢查 Wiki，找出彼此矛盾的論點、過時的資訊、沒有連結的孤兒頁面，甚至讓 AI 提示你接下來該尋找什麼新資料。
    

#### 4. 為什麼這行得通？

傳統知識庫（如個人 Wiki）之所以容易失敗，是因為「維護成本（打標籤、連檔案、改錯）太高」，人類容易感到厭煩。而 **LLM 擅長做瑣碎的文書管理工作，且不知疲倦**。人類只需負責「提供資料、引導方向、深度思考」，瑣碎的整理全交給 AI。

### 三、 如何運用（Application Guide）

作者在文中提到他自己的實踐方式：**一邊開著 LLM Agent（如 Claude Code 或 OpenAI Codex），另一邊開著筆記軟體 Obsidian。** AI 在後台改檔案，他在 Obsidian 裡即時看圖表與連結的變化。

你可以將此模式運用在以下場景：

#### 1. 具體應用場景

- **深度的學術/主題研究：** 連續幾週或幾個月閱讀大量論文與報告，讓 AI 幫你建立一個理論演進的脈絡網絡。
    
- **個人成長與健康管理：** 匯入日常日記、健檢報告、Podcast 筆記，建立個人的心理與身體健康演進模型。
    
- **讀書精讀筆記：** 讀長篇巨著（如小說、歷史書）時，每讀完一章就匯入，讓 AI 自動建立人物誌、時間軸、主題連結（類似魔戒或權力遊戲的粉絲維基）。
    
- **企業與團隊知識庫：** 將 [[Slack]] 討論串、會議逐字稿、客戶反饋餵給 AI，由 AI 自動維護公司內部 Wiki，保證資訊永遠最新。
    

#### 2. 實踐步驟與推薦工具

1. **準備一個 Markdown 筆記軟體：** 推薦使用 **Obsidian**（因為它支援雙向連結、擁有 Graph View 關係圖，且檔案都是純文字的 Markdown）。
    
2. **複製這份精神給你的 AI Agent：** 這篇 Gist 本身就是一個「概念提示詞（Prompt）」。你可以把它複製貼給具備檔案讀寫功能的 AI Agent（例如 Claude Code、Cursor、或是寫好自訂指令的 GPTs），跟它說：「我們要一起打造這樣的系統。」
    
3. **編寫你的規範文件（Schema）：** 與 AI 共同討論出一份規則（例如定義資料夾結構、命名要用英文還是中文、標籤規格），記錄在主目錄。
    
4. **搭配實用工具：**
    
    - **Obsidian Web Clipper：** 在瀏覽器看到好文章，一鍵轉成 Markdown 存入原始資料夾。
        
    - **Marp 外掛：** 直接用 Markdown 語法生成簡報，方便把 Wiki 的研究成果直接變成報告投影幕。
        
    - **Dataview 外掛：** 讓 AI 在筆記上方加上 YAML 屬性欄位（如日期、標籤），可以用類似 SQL 的方式動態撈出資料表。
        

### 總結

這篇文章提供了一種「把知識庫當作程式碼（Codebase）來維護」的新思維：你不用再自己動手寫筆記、拉連結，你只要扮演「架構師與總編輯」，讓 AI 擔任「程式設計師」，幫你不知疲倦地編織與擴展你的大腦外掛。

---
在 Andrej Karpathy 的這套構想中，AI Agent **不會完全自己憑空**決定一個新專案名稱，而是會根據你丟給它的**原始資料內容**或**研究主題**，自動提議並建立一個合適的專案（在 Wiki 中通常表現為一個「主資料夾」或「核心索引頁面」）。

更精準地說，它的運作邏輯與人機協作方式如下：

### 1. 專案的「起點」是由你（或資料）發起的

當你開始一個全新領域的研究時，你不需要手動去建一堆資料夾，你只需要做兩件事之一：

- **你對 AI 說：** 「我最近要開始研究『固態電池技術的最新進展』，這是我收集的第一批論文。」
    
- **你直接丟入資料：** 你把 5 篇關於「Vision Transformer (ViT)」的論文直接丟進原始資料夾。
    

### 2. AI 如何自動建立專案與結構？

一旦接收到新主題的資料，AI Agent 會執行以下自動化步驟：

- **自動命名與建立核心頁面（Index Page）：** AI 閱讀完資料後，會自動在 Wiki 中建立一個新頁面（例如 `Vision-Transformer-Study.md` 或 `Solid-State-Battery.md`），並將其作為這個專案的「錨點（Anchor）」。
    
- **自動分類與打散資訊：** 它不會只把這專案當成一個獨立死板的資料夾。它會掃描整個知識庫，看看這個新專案和以前的舊知識有沒有關聯。
    
    - _範例：_ 發現固態電池的某些化學原理與你半年前研究的「鋰電池降解」有關，它會自動跑去舊的專案頁面，加上一行：`[[Solid-State-Battery]] 中提到的新材料解決了這個降解問題`。
        

### 3. 如何在規範（Schema）中約束 AI 的命名？

為了防止 AI 自己亂取名字（例如今天叫 `AI-Note`，明天叫 `Artificial_Intelligence_Review`），Karpathy 強調一定要有 **`Schema`（規範文件）**。

你可以在規範中限制 AI：

- 「所有新專案的主頁面命名，一律使用 `[主題名稱]-Project-Index` 的格式。」
    
- 「專案資料夾一律使用小寫與連字號（如 `quantum-computing`）。」
    

### 💡 結論：你是「總編輯」，AI 是「執行編輯」

它會自動幫你開立專案名稱、建立檔案、串聯超連結，但**最後的決定權在於你**。

在 Obsidian 的畫面中，你會看到左側目錄突然多出了一個 AI 幫你建好的新專案資料夾與脈絡圖。如果你覺得它取的名字不夠精準，你隨時可以手動修改該頁面名稱，而 AI 隨後也會自動把整個知識庫裡所有連到該頁面的「雙向連結」全部自動更新。
