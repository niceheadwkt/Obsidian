在 Ollama 的環境中，`ollama launch codex-app` 與 `ollama launch codex` 這兩個選項的核心差異，在於它們所對應和配置的 **OpenAI Codex 工具介面與工作型態（軟體載體）不同**。

簡單來說：

- **`codex-app`** 控制的是**桌面獨立應用程式（圖形介面）**。
    
- **`codex`** 控制的是**終端機命令列工具（CLI）**。
    

這兩個工具的詳細差異與特性如下：

### 1. `ollama launch codex-app`（桌面 App 模式）

這個選項是用來配置並啟動 OpenAI 官方發布的 **Codex 桌面應用程式（Desktop App）**。

- **操作介面**：完整的圖形化介面（GUI），像一個獨立的專案管理中心。
    
- **核心功能**：
    
    - **多任務與多 Agent 並行**：可以在同一個專案中同時開啟多個執行緒（Threads），讓多個 AI 代理同時處理不同的編碼任務。
        
    - **內建瀏覽器與標註**：App 內建瀏覽器，當你在開發網頁時，可以直接在畫面上進行視覺化標註，讓 AI 根據你的標註直接修改前端程式碼或樣式。
        
    - **程式碼審查模式（Review Mode）**：擁有直觀的 Diffs 對比介面，讓你在合併程式碼前可以一行行檢查、留言和修改。
        
- **Ollama 的角色**：執行此命令時，Ollama 會自動將 Codex 桌面 App 的 API 端點修改為本機的 OpenAI 相容端點（或 Ollama Cloud），讓你完全**免去 OpenAI 的付費訂閱**，直接用本機的開源模型（如 `qwen3`、`gemma4` 或 `llama3`）來驅動這套圖形化 Agent。
    

### 2. `ollama launch codex`（命令列 CLI 模式）

這個選項則是配置與管理 OpenAI 官方的 **Codex CLI 終端機工具**。

- **操作介面**：完全在終端機（Terminal）環境下運作，透過命令列（`codex`）或搭配 IDE（如 VS Code 擴充套件）隱藏在背景運作。
    
- **核心功能**：
    
    - **終端機直接操作**：適合喜歡完全在文字介面、Git 工作流或自動化腳本（如 CI/CD 流水線）中工作的開發者。
        
    - **本機工作目錄與 Sandbox**：透過終端機讀取、修改並直接在你目前的目錄執行程式碼（並整合 OS 級別的 Sandbox 沙盒防護）。
        
- **Ollama 的角色**：配置並提供本地模型（預設常搭配專為程式碼設計的開源模型如 `gpt-oss` 等）給本地終端機的 `codex` 指令呼叫。
    

### 關鍵差異對比

|**特性**|**ollama launch codex-app**|**ollama launch codex**|
|---|---|---|
|**載體型態**|視窗應用程式（macOS / Windows App）|終端機工具（CLI / `npm i -g @openai/codex`）|
|**適合場景**|網頁前端開發、複雜專案的多任務管理、程式碼視覺化 Diff 審查|腳本自動化、終端機重度使用者、CI/CD 整合、後端純文字重構|
|**獨有特色**|內建瀏覽器標註修改、多執行緒並行 Agent 視窗|輕量化、資源佔用極低、易於與其他 CLI 工具鏈串接|
|**設定檔隔離**|獨立儲存於 `codex-app` 的 Profile 中|獨立儲存於 `codex`（CLI）的 Profile 中|

> 💡 **使用建議**：
> 
> 如果你正在寫網頁、需要直觀地看 AI 修改了哪些檔案的 Diff，或是想邊看網頁畫面邊指引 AI 修正，請選擇 **`codex-app`**；如果你日常開發都在 Vim、Tmux 或純 Terminal 裡，只需要 AI 當作背景的指令執行工具，請選擇 **`codex`**。兩者的設定檔是獨立分開的，所以即使切換模型也不會互相影響。

在針對 **「專案管理與協作方式」** 進行比較前，首先要釐清一個技術核心：**`ollama launch codex-app` 本質上並不是另外開發一套獨立的 App，而是用來「引導、設定並啟動」你電腦裡那套 Codex 桌面應用程式（Codex Desktop App）的加速指令。**

你可以把它想像成：Ollama 扮演了「大腦與後勤總管」，而 Codex 桌面 App 是「操作介面與四肢」。

這兩者在處理「專案」時的操作差異，主要體現在 **核心模型驅動（算力來源）**、**安全沙盒（Sandbox）範圍** 以及 **專案初始化配置** 上。以下為您梳理這兩種操作路徑在專案模式下的實務差異：

### 1. 專案環境初始化與模型繫結

當您在專案中準備讓 AI 介入開發時，啟動方式決定了專案與模型的綁定深度。

- **直接點擊 Codex 桌面應用程式：**
    
    - **雲端優先（預設）**：專案預設會連結到 OpenAI 的雲端 API。每次進行專案層級的程式碼理解、Index（索引）建立時，都需要消耗雲端 Token 與扣款。
        
    - **手動配置**：若要在專案內改用本地模型，您必須進到 App 的 `Settings -> Models -> Custom Endpoint`，手動輸入 `http://localhost:11434/v1`，並手動輸入相容的 Model Name（例如 Qwen 或 Llama）。
        
- **使用 `ollama launch codex-app` 指令：**
    
    - **一鍵本地化專案**：在終端機切換到您的專案目錄（例如 `cd /path/to/my-project`），執行該指令後，Ollama 會自動在背景完成 API 轉發（Forwarding）與環境變數注入。
        
    - **指定模型專案**：可以直接下達 `ollama launch codex-app --model qwen3`，它會自動將這個專案的預設驅動大模型鎖定在本機的特定模型，App 開啟後直接就是完全離線、免付費的專案開發狀態。
        

### 2. 專案索引（Project Indexing）與 Token 消耗

對於多檔案、有複雜目錄結構的專案，AI 需要先「讀懂」整個專案結構。

- **直接點擊 Codex 桌面應用程式：**
    
    - 當你匯入專案資料夾時，App 的 Agent 會開始掃描代碼。若使用雲端模型，大型專案（含許多 legacy code 或大檔案）在建立 [[RAG]]（檢索增強生成）索引時，可能會因為 Context Window 的限制或雲端費率，導致專案初始化成本極高。
        
- **使用 `ollama launch codex-app` 指令：**
    
    - 此模式下，專案的向量化（Embedding）與本地程式碼解析會直接交給本機的 Ollama 處理。
        
    - 這意味著，無論您的專案有多少個資、多少舊系統（Legacy System）的 JSP、Java 或 SQL 檔案，**所有專案代碼的掃描與分析都在您本機的記憶體與顯卡中完成，完全不會有企業敏感原始碼外洩到外網的風險**，且完全免費。
        

### 3. 多 Agent 協作（Multi-Agent Workspace）的操作彈性

Codex 桌面應用程式最強大的地方在於以「專案」為核心的多執行緒（Threads）協作。

- **直接點擊 Codex 桌面應用程式：**
    
    - 在 GUI 介面中，你需要手動為專案建立不同的 Agent（例如：一個負責寫前端、一個負責重構後端、一個負責 Review 程式碼）。如果想切換不同的模型來測試同一段程式碼的修改效果，必須在 UI 選單中反覆點選切換。
        
- **使用 `ollama launch codex-app` 指令：**
    
    - 支援在啟動時，直接透過參數為專案指派不同的 Profile。例如：
        
        - `ollama launch codex-app --profile backend`（自動配置高邏輯推理的模型處理後端）
            
        - `ollama launch codex-app --profile frontend`（配置適合網頁前端與 CSS 的模型）
            
    - 這讓您在終端機就能以自動化腳本的方式，針對同一個專案開啟多個相互獨立、配置不同的 Codex 桌面視窗。
        

### 專案模式操作對比總結

|**專案操作面向**|**直接開啟 Codex 桌面 App**|**透過 ollama launch codex-app**|
|---|---|---|
|**專案隱私與安全**|預設上傳雲端（需注意公司資安資產規範）|**100% 本地端**，代碼不出本機，適合內部企業系統開發|
|**大檔案/舊代碼掃描**|依雲端費率計價，可能產生高額 Token 費用|運用本地算力，**無限制掃描與索引建立**|
|**多專案切換彈性**|需在 App 介面內頻繁手動切換專案與模型設定|在對應的專案目錄下一鍵啟動，**自動綁定該專案專屬的本地模型**|
|**API 串接手續**|必須在 App 內的設定選單手動填寫 Endpoint|**全自動化**，Ollama 會在背景幫 App 與本地模型搭好橋樑|

> 💡 **最佳實務建議**：
> 
> 如果您今天要經手的是**公司內部的敏感專案（例如涉及資料庫機密、ERP 邏輯、商業隱私）**，強烈建議養成從終端機使用 `ollama launch codex-app` 的習慣來開啟專案。這樣可以確保 Codex 桌面應用程式那漂亮的 UI、多 Agent 協作、以及視覺化 Diff 審查功能完好保留的同時，底層的核心大腦卻是完全安全、離線且免費的本機模型。

在 Windows 環境下，**「初始化與首次開啟」確實仍然需要透過 PowerShell（或 CMD、VS Code 內建終端機）來下達指令。**

因為像 `ollama launch codex-app` 這類延伸的整合指令，核心邏輯是透過命令列介面（CLI）去偵測你目前所在的「工作目錄」，並自動在背景處理模型端點（Endpoint）的橋接。

不過別擔心，這並不意味著你每次寫程式都得開著一個黑畫面的視窗。以下提供你最順手、最符合開發習慣的兩種操作專案流程：

### 方法一：直接在 VS Code（或其他 IDE）內建終端機執行（最推薦）

如果你平常習慣用 VS Code 開發，根本不需要另外打開一個獨立的 PowerShell 視窗，直接在編輯器裡一氣呵成即可：

**1.用 VS Code 開啟你的專案資料夾：**步驟 1。

點擊 VS Code 的 `檔案 -> 開啟資料夾`，選取你的專案（例如你的 Python 或 Java 專案目錄）。

**2.開啟內建終端機：**步驟 2。

按下快速鍵 `Ctrl + ~`（或點選上方選單的 `終端機 -> 新終端機`）。此時終端機預設就會直接定位在你目前的專案目錄下。

**3.輸入 Ollama 啟動指令：**步驟 3。

在終端機內直接輸入指令並送出：

PowerShell

```
ollama launch codex-app --model qwen3
```

_(註：`--model` 後面請帶入你本機已經 `ollama pull` 下來的程式碼模型名稱。)_

此時，**Codex 桌面應用程式（GUI 視窗）就會自動彈出來**，並且它會自動把 VS Code 目前這個專案的目錄載入進去，底層算力也已經綁定好你的本地大模型。

### 方法二：寫成 Windows 批次檔（.bat），以後「點兩下」直接開專案

如果你不想每次都打字，可以幫特定的常駐專案建立一個專屬的「捷徑腳本」，連 PowerShell 都不用開。

1. 在桌面或專案資料夾滑鼠右鍵 -> `新增 -> 文字文件`。
    
2. 將副檔名改為 `.bat`（例如：`開啟我的AI專案.bat`）。
    
3. 點擊右鍵編輯它，填入以下兩行程式碼後儲存：
    
    DOS
    
    ```
    @echo off
    cd /d "C:\您的專案真實路徑\my-project"
    ollama launch codex-app --model qwen3
    ```
    

```
4. **以後只要在桌面上對這個 `.bat` 檔案點兩下**，它就會自動幫你切換目錄並呼叫 Ollama，把 Codex 桌面 App 帶到你面前。

### 補充說明：為什麼不能直接滑鼠雙擊「Codex Desktop App」圖示？

如果你直接去點桌面上的 Codex App 圖示，它不知道你現在想處理哪一個專案（資料夾），而且它的底層會預設去連 OpenAI 雲端的 API，要求你輸入 API Key 或扣你的雲端儲存點數。

透過 PowerShell 或上述的 `.bat` 檔案執行 `ollama launch`，其實是在幫你做兩件事：
1. **告訴 Codex App**：「請直接載入我現在這個資料夾當作工作區（Workspace）。」
2. **劫持 API 路由**：「把雲端連線切斷，把所有 AI 請求轉彎送到本機的 `127.0.0.1:11434`（Ollama）。」

所以，透過命令列來做「初始化啟動」，能省去你在圖形介面裡繁瑣的設定手續！
```

要在 Codex 桌面應用程式（Codex Desktop App）中確認它目前是否「真的」由本機的 Ollama 模型驅動，而不是偷偷連回雲端，您可以從介面的兩個地方進行驗證：

### 1. 檢查主視窗左下角（或右上角）的模型狀態列

當您透過 `ollama launch codex-app` 成功啟動後，Codex 桌面應用的狀態列會顯示當前使用的大模型名稱。

- **雲端模式下**：通常會顯示 `gpt-4o`、`claude-3-5-sonnet` 等 OpenAI 或 Anthropic 的官方模型名稱。
    
- **本地 Ollama 模式下**：您會看到顯示為您指定的本地模型名稱（例如 **`qwen3`**、**`llama3`**、**`gemma4`** 等）。
    
- 部分版本在模型名稱旁邊還會特別註記 **`Local`** 或 **`Custom Endpoint`** 的標籤，這就代表它正透過本地端點運作。
    

### 2. 查看設定選單中的 API 端點（Settings）

您可以直接進入 Codex App 的底層設定檔檢查路由：

1. 在 Codex 桌面應用程式中，點擊左下角的 **齒輪圖示（Settings）**。
    
2. 切換到 **Models** 或 **Providers** 分頁。
    
3. 找到 **OpenAI Compatible** 或 **Custom Endpoint** 區塊。
    
4. 檢查其中的 **Base URL（API 網址）**：
    
    - 如果它是綁定本地 Ollama，該網址一定會被自動修改為：`http://localhost:11434/v1` 或 `http://127.0.0.1:11434/v1`。
        
    - **API Key** 欄位通常會變成灰色、顯示 `ollama`、或直接被留空。
        

### 3. 最直觀的方法：看您的 PowerShell / 終端機反應

除了看 Codex App 的介面，最騙不了人的其實是您剛剛用來啟動的那個 **VS Code 內建終端機**（或 PowerShell）。

當您在 Codex 應用程式的聊天視窗中輸入一對話，或者讓它幫您重構一段專案程式碼時：

- **請盯著 VS Code 的終端機畫面**：您會看到終端機開始瘋狂滾動**本地 API 請求日誌（Log）**，類似於：
    
    Plaintext
    
    ```
    [GIN] 2026/06/01 - 10:15:30 | 200 |  1.24s |       127.0.0.1 | POST     /v1/chat/completions
    ```
    
- 這代表 Codex App 每產生一個字，都是即時向您本機的 Ollama 發送 `POST` 請求。如果放任它閒置，終端機的 Log 就會停止滾動。
    

只要看到終端機有隨著您的操作跑出本地連線日誌（127.0.0.1），並且模型名稱符合，就能百分之百確定您的專案代碼正安全地待在本地，由您的顯卡與 Ollama 在全權處理了！
