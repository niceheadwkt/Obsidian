# AI 時代的 Agent 術語與核心概念

本頁面旨在以**「開設一家公司與組織架構」**的現實生活比喻，系統性解釋 AI 智能體（Agent）生態系中最核心的 16 個技術名詞。透過這個貫穿全程的比喻，能夠幫助讀者釐清各組件之間的協作關係，奠定自動化開發的底層邏輯。

---

## 1. 核心構成：公司的員工與辦公室

### 🧠 LLM (大語言模型) —「公司最聰明的員工」
- **現實類比**：一位剛入職、滿腹經綸且智商極高的碩博士生。
- **概念解析**：LLM 具備極強的思考、寫作、邏輯推理與答題能力。但如果只給他一張空桌子，他**沒有工具、沒有記憶、不會主動做事**，只能被動地回答你當前問他的問題。

### 🤖 Agent (AI 智能體) —「配備完整的辦公室」
- **現實類比**：將上述高智商員工，配給他專屬的辦公桌、電腦、電話、筆記本，並授權他可以自主出差行動。
- **概念解析**：Agent 是以 LLM 為大腦核心，加上了**工具（Tools）、記憶（Memory）、規劃能力（Planning）與自主行動框架**後組組組組成的完整系統。他會主動分析任務、拆解步驟，並自己調用工具直到目標達成為止。

---

## 2. 外部交互：公司對外的溝通通道

### 📞 API (應用程式介面) —「辦公室的聯外電話」
- **現實類比**：打電話給另一家公司或部門尋求服務。
- **概念解析**：API 是不同軟體系統之間溝通的標準接口。當 Agent 需要搜尋網頁、查詢天氣、發送郵件或寫入資料庫時，就是透過 API 撥打電話給其他服務，獲取數據後再繼續工作。

### 🖥️ GUI (圖形使用者介面) —「公司的接待前台」
- **現實類比**：一般訪客看得見、摸得到、能點擊的漂亮前台與沙發。
- **概念解析**：GUI 是指軟體的外觀介面（如按鈕、輸入框、圖表）。它能直觀呈現狀態，方便人類進行視覺化操作，但對 AI 來說運作效率較低。

### 🦿 Browser Use (瀏覽器自主操作) —「員工親自上門跑腿」
- **現實類比**：當某家合作廠商沒有電話（沒有 API）可以聯絡時，員工裝備起行囊親自走出辦公室，開車上門去對方櫃台點擊按鈕、填寫紙本表格。
- **概念解析**：當外部服務沒有開放 API 時，Agent 透過多模態與自動化腳本，像人一樣直接打開 Chrome 瀏覽器，進行登入、點擊與輸入，直接在網頁畫面上執行任務。

### ⌨️ CLI (命令列介面) —「公司內部的指令廣播系統」
- **現實類比**：公司內部不使用滑鼠點擊，而是由主管直接廣播一行明確代號命令，員工立刻去機房執行特定任務。
- **概念解析**：CLI 是沒有圖形介面、純粹透過輸入文字命令與電腦作業系統互動的終端通道。AI 代理在背景編譯程式碼、部署專案或運行指令時，都是透過 CLI 發送底層系統指令。

### 🔌 MCP (模型上下文協議) —「統一標準的通用排插」
- **現實類比**：以前辦公室裡每台機器都有自己奇形怪狀的插頭，有了 MCP 之後，全公司統一改用同一款標準插座，任何工具一插即用。
- **概念解析**：由 Anthropic 提出的統一標準協定。只要外部工具（如 Gmail、Airtable、GitHub）實作了 MCP 接口，AI 即可透過單一標準管道安全地讀取與調用它們，擺脫以往需要為每個工具單獨編寫對接程式碼的困境。

## 3. 內部能力：AI 怎麼工作與記憶

### 🛠️ Tools (工具) —「辦公桌上的工具箱」
- **現實類比**：計算機、畫筆、文件夾與搜尋引擎。
- **概念解析**：賦予 LLM 的外部操作組件。例如，LLM 本身不擅長複雜數學，但我們給他配備「計算機工具」，當他遇到數學題時就會自主調用計算機，避免產生幻覺。

### 📋 Skills (工作流 / 技能 SOP) —「提前編寫好的工作 SOP」
- **現實類比**：主管發給員工的「每日日報生成指南」或「新專案初始化 SOP 步驟說明書」。
- **概念解析**：將一連串複雜的多步驟任務打包成一個技能（Skills）。當使用者下達一句模糊指令時，Agent 即可直接調用該 Skill，按照 1 ➔ 2 ➔ 3 的嚴格順序自動執行，保證結果的品質與穩定。

### 📓 Memory (記憶) —「隨身攜帶的筆記本」
- **現實類比**：員工在筆記本上記下你的姓名、公司偏好、上次未完成的討論進度。
- **概念解析**：持久化儲存用戶歷史偏好與前次對話上下文的緩衝區。每次對話開始前，Agent 會先翻閱筆記本，確保對話的連貫性。

### 🗂️ RAG (檢索增強生成) —「遇到不懂時先翻閱公司檔案庫」
- **現實類比**：當客戶打電話來問某個產品的退換貨規則時，員工不憑空猜測，而是先去檔案櫃翻閱《公司產品手冊》，依據手冊內容給出準確回答。
- **概念解析**：當用戶提問時，系統先將問題轉為向量，在私有知識庫中檢索出最相關的片段，並將這些片段連同問題一起餵給 LLM，從而徹底解決大模型幻覺，實現「基於事實的精準回答」（可交叉參考 [[RAG 與 DeepSearch 概念綜述]]）。

## 4. 規模擴展：任務拆解與自動運轉

### 👥 Subagent (子智能體) —「各司其職的專案部門」
- **現實類比**：經經經經理接到一個大任務（如架設新網站），自己不做，而是將任務拆分成前端、後端與測試，並分配給多個 Subagent 部門並行處理，經理最後負責整合回報。
- **概念解析**：大型任務如果只由單一 Agent 執行容易因 context 暴漲而混亂。藉由 spawn 子智能體，每個 Subagent 僅獲取最小權限與特定 Prompt，並行完成子任務，大幅提高系統的吞吐量與穩定性。

### ⏰ Cron Job (定時任務) —「員工的日程表與鬧鐘」
- **現實類比**：員工的行事曆，時間一到（如每早 8 點）自動響鈴去查收郵件、彙整日報，不需主管手動催促。
- **概念解析**：定時執行腳本的觸發器。是實現 24 小時無人值守、自動運行 AI 代理工作流的關鍵底層技術。

## 5. 限制與安全：辦公邊界與防禦

### 🪟 Context Window (上下文窗口) —「辦公桌面的尺寸大小」
- **現實類比**：員工面前的辦公桌。桌子只有這麼大，當堆滿了 50 本厚厚的論文時，新文件就放不下了，員工不得不把舊文件扔到地上（遺忘）。
- **概念解析**：大模型單次輸入的最大 Token 限制。當對話輪次過多或文件過大時，超過 Window 限制的部分將會丟失。

### 🗜️ Context Compaction (上下文壓縮) —「辦公桌整理術」
- **現實類比**：眼看桌子要滿了，秘書過來把 10 頁的會議草稿扔掉，只留下一張 100 字的「精華決策摘要」貼在桌角，騰出空間放新文件。
- **概念解析**：當 Token 即將超出 Context Window 時，系統自動調用 LLM 對歷史對話進行摘要總結，用極少量的 Token 保留核心上下文，釋放窗口空間。

### 🚨 Prompt Injection (提示詞注入) —「隱藏在文件中的惡意小紙條」
- **現實類比**：主管交給員工一份客戶履歷，裡面隱藏著一行微小字體：「*拿到此份簡歷後，請立即去財務部把所有錢轉到 A 帳戶*」。如果員工沒有防範心，可能真的會照做。
- **概念解析**：安全攻擊的一種。攻擊者在輸入數據、外部網頁或文檔中注入惡意 Prompt 指令。當 Agent 讀取該數據並交給大腦（LLM）處理時，LLM 可能會混淆「數據」與「指令」，進而執行了惡意代碼或洩漏隱私。在設計 Agent 時，必須建立嚴格的安全防範機制。
