# 個人知識管理與 LLM Wiki 構築系統 (PKM & LLM Wiki System)

本頁面整合了基於 Andrej Karpathy 理念的 **LLM Wiki 知識複利架構**，以及 **Obsidian 雙向連結 (Backlinks)** 的底層技術與四大實務應用場景，旨在協助您打造一個能自主演進的「外部大腦」。

---

## 1. LLM Wiki 模式：AI 驅動的知識庫維護

傳統的知識庫（如個人 Wiki）最終往往被人類放棄，因為「手動連檔案、修錯、分類與記帳」的**維護成本太高**。而 LLM Wiki 的核心思想在於將這些繁瑣的Bookkeeping工作交給 AI Agent。

### 三層架構 (Architecture)
1. **原始資源層 (`sources/`)**：不可變的論文、網頁剪輯、逐字稿等。此層為**唯讀**，是事實的單一來源。
2. **Wiki 知識層 (`wiki/`)**：由 AI 全權寫入、維護的 Markdown 文件，包含概念、實體、比較分析。
3. **規範層 (`schema.md` / `CLAUDE.md`)**：寫給 AI 遵循的 SOP 與命名規範文件。

### 知識「編譯」與累積
與每次問答都需要重新搜尋碎紙片的傳統 RAG 系統不同，LLM Wiki 採用**增量編譯**：
- 丟入新文獻時，AI 會主動閱讀、在 `wiki/` 建立摘要，並自動更新可能相關的 10~15 個現有 Wiki 頁面，標記衝突並拉起雙向連結。
- 查詢時，AI 給出的高價值綜合分析，會被**重新寫回 Wiki 作為新頁面**，使對話歷史沉澱為永恆知識。

### Karpathy 的 LLM Wiki 實作指引
依據 [[raw/Andrej Karpathy 方法的 LLM Wiki 知識庫.md|Andrej Karpathy 方法的 LLM Wiki 知識庫]] 實務，搭建該知識庫包含以下四大核心步驟：
1. **準備本地環境**：
   - 使用 **Obsidian** 作為本地 Markdown 筆記編輯與雙向連結管理器。
   - 建立本地專案資料夾作為知識倉庫，其推薦之內部目錄結構為：
     - `sources/`：放置原始不可變資料（如論文、文章、會議逐字稿等）。
     - `wiki/`：放置 LLM 自動編譯與更新維護的 Wiki 筆記檔案。
     - `inbox/`：存放待處理的原始或臨時筆記。
2. **定義知識結構 (Schema)**：
   - 於專案根目錄配置 `schema.json`（或類似規範文件如 `schema.md`）。
   - 在其中預先定義您希望追蹤的核心概念、實體或主題，作為 AI 代理分類整理的藍圖。
3. **配置 LLM 代理 (Agent)**：
   - 推薦使用 **Claude Code** 作為直接存取本地檔案並執行編譯與整理工作的 AI 代理。
   - 配置代理的工作提示詞（Prompt SOP），使其工作流包含：掃描新資料 $\rightarrow$ 閱讀 schema 規範 $\rightarrow$ 將新資料編譯整理至 `wiki/` 並建立 `[[雙括號]]` 交叉引用。
4. **運作與維護**：
   - **編譯新知**：隨時將新資訊放入 `sources/`，下令代理執行編譯整理。
   - **互動查詢**：直接在終端機啟動 Claude Code 進行知識庫問答，代理會自動跨文件檢索以給出立基於您專屬知識庫的回答。
   - **迭代優化**：定期檢視並修訂 schema，視需求增刪追蹤主題。

## 2. 雙向連結 (Bi-directional Linking) 技術指南

**雙向連結 (Backlinks)** 是 Obsidian 的靈魂。當您在筆記 A 連結到筆記 B，系統會自動在筆記 B 建立反向引用連結，編織出大腦般的神經網路。

### 語法與應用技巧
- **基本引用**：使用 `[[頁面名稱]]`。若目標頁面不存在，Obsidian 會預留該結構，點擊即可自動創建新筆記。
- **設定別名**：使用 `[[頁面名稱|顯示別名]]`。在閱讀模式下只顯示別名，但點擊依然跳轉至核心筆記，使文筆語氣流暢。
- **精準標題引用**：使用 `[[頁面名稱#標題名稱]]`。在引號內鍵入 `#` 即可自動跳出所有標題供選取。
- **精細化段落引用**：使用 `[[頁面名稱^段落代碼]]`。鍵入 `^`（Shift + 6）並搜尋關鍵字，Obsidian 會在目標段落後自動生成隨機代碼（如 `^8a3b2c`）進行精確引用。
- **自動顯示反向連結設定**：
  進入「設定」 $\rightarrow$ 「核心外掛程式」 $\rightarrow$ 開啟「反向連結 (Backlinks)」 $\rightarrow$ 點擊齒輪開啟「在文件內顯示反向連結」。這樣每篇筆記底部都會自動呈現「誰引用了我」。

## 3. 四大場景應用實務

藉由雙向連結，您可以將以下日常場景串接，產出強大的知識圖譜：

### 📚 讀書筆記
- **操作**：記錄書中的核心概念時，直接包上雙括號（例如 `[[複利效應]]`）。
- **效果**：當累積了數十本讀書筆記後，打開「複利效應」頁面，底部的反向連結會瞬間呈現《原子習慣》、《財富自由主義》等書籍的關聯性，協助跨書本融會貫通。

### 💻 程式學習
- **操作**：為特定的演算法、語法特性或常見 Error Code 建立獨立頁面（例如 `[[WinError 2]]`、`[[Transformer 運作原理]]`）。
- **效果**：每次寫練習筆記或 Bug 排除日誌時進行關聯。往後複習時，能快速點進該技術頁面，查看所有曾出現過此技術的實際程式碼專案。

### 💼 工作專案
- **操作**：為每個專案、會議、利害關係人建立卡片（例如 `[[Tomcat 部署專案]]`、`[[專案經理 A]]`）。
- **效果**：在每日日記中引用專案與人名。當需要評估專案進度或準備 1-on-1 會議時，點進 `[[Tomcat 部署專案]]`，即可看到與此專案相關的所有歷史會議摘要、待辦清單與決策變更日誌。

### 📱 App 開發與系統設計
- **操作**：為系統架構、API 接口、數據庫 Schema 建立關聯網絡（例如 `[[mtStructs.xml]]`、`[[DAO/VO 模式]]`）。
- **效果**：當 AI 代理或您修改系統組件時，能快速釐清 Controller 與 JSP 的對應關係，避免牽一髮而動全身的配置錯誤。

## 4. 工具對比：Obsidian vs. Roam Research

- **Obsidian**：
  - **優點**：本地優先（純 Markdown 檔案存在自己電腦），隱私與安全性高；社群插件極其龐大（如 Dataview 進行 frontmatter 查詢，Marp 生成簡報）；免費版即極其強大。
  - **缺點**：多裝置同步需手動配置（或使用付費的 Obsidian Sync）。
- **Roam Research**：
  - **優點**：雲端大綱型 (Outliner) 始祖，每個 Bullet block 都是獨立實體，塊級引用 (Block Reference) 性能極佳。
  - **缺點**：必須付費訂閱，且資料存在雲端，對內部敏感代碼或資安要求高的專案不友善。
