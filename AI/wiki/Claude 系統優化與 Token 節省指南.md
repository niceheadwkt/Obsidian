# Claude 系統優化與 Token 節省指南

**Claude 系統優化與 Token 節省指南** 旨在解決開發者與高頻用戶常面臨的「Tokens 消耗過快」、「頻繁達到用量上限（Usage Limit）」等成本與效率問題。本篇整理了底層 Token 暴漲原因、官方針對 Claude Code 的節流建議、模型定價與選用策略，以及 `CLAUDE.md` 的優化準則。

---

## 1. Token 暴漲的底層原因與優化原則

### 🔍 為什麼 Token 消耗會呈指數型增長？
在大模型的運作機制中，**每次對話並非獨立發生**。為了維持對話連貫性，每次您送出新訊息時，Claude 都會將「`先前的對話歷史` + `載入的上下文文件` + `您的新 Prompt`」打包，作為一個巨大的 Input 重新傳送至雲端進行推理。
因此，若您在同一個 Session（對話）中丟入大檔案並進行了 20 次對話，該大檔案會被重複計費並讀取 20 次，導致 Tokens 瞬間燒光。

### 🛠️ 4 個常見用量超限情境與縮減技巧

1.  **情境一：對話累積過長**
    *   **技巧**：一旦確認某個代碼 Bug 已修復或特定大綱已定稿，請立即開新對話（New Session），不要在同一個對話中無限延伸。
2.  **情境二：直接上傳超大 PDF 文獻**
    *   **技巧**：避免將數百頁的書籍或論文直接整本丟給 AI。先用工具將 PDF 切割成關鍵章節，或僅複製出與當前任務直接相關的文字段落。**光是將 PDF 進行結構化分段處理，Token 消耗量就可相差達 22 倍**。
3.  **情境三：反覆重寫整個代碼檔案**
    *   **技巧**：在修改代碼時，不要讓 AI 盲目地將整個檔案重新輸出。在 Prompt 中明確要求：「請僅輸出需要修改的代碼區間（Diff 格式）或特定 Function 即可」。
4.  **情境四：冗餘的 System Prompt 與背景資訊**
    *   **技巧**：遵循「少即是多」原則，剔除 Prompt 中那些客套、重複、或無關痛癢的背景描述，將指令聚焦於核心任務。

## 2. Claude Code (終端機 CLI) 官方 4 條節流建議

當您在本地使用 [[Claude|Claude Code]] CLI 工具時，其高頻的檔案掃描與指令執行容易在不知不覺中消耗大量額度。官方給出以下 4 條防禦建議：

1.  **限制終端機輸出長度**：避免在對話中執行如 `git log`、`npm test` 等會產生上千行文字的命令。建議使用 `git log -n 5` 或過濾特定錯誤訊息，防止大量終端文字被打包成上下文發送。
2.  **配置排除清單 (.gitignore)**：確保將 `node_modules/`、`build/`、`dist/` 以及像 `.obsidian/workspace.json`（參見 [[Git GUI 與 GitHub 雙向同步實務]]）等隨時變動的環境暫存檔寫入排除名單，防止 Claude Code 在掃描專案目錄時將其完整讀取。
3.  **適時使用 `/clear` 命令**：當前任務（如完成一個 Feature 開發）結束後，於終端機輸入 `/clear` 清空當前對話記憶，以乾淨的上下文開始下一個 Bug 的修復。
4.  **精確指引檔案範圍**：若專案很大，在要求 Claude Code 編輯或讀取時，指明具體檔案路徑與行數（如：`請幫我修改 src/utils.py 的第 45 行至 60 行`），避免 AI 掃描不相關的模組。

## 3. 模型選用策略：Opus vs. Sonnet vs. Haiku

選擇合適的模型是控制成本的核心。若盲目使用最強的模型處理簡單任務，**可能會多花費高達 5 倍的 Token 成本**：

| 模型名稱 | 推理能力 | 速度 | 價格比 | 適合場景 |
| :--- | :--- | :--- | :---: | :--- |
| **Claude 3.5 Sonnet** | 極強（代碼能力首選） | 快 | 中 | **主力開發模型**。適合程式撰寫、複雜語意推理、系統整合。 |
| **Claude 3.7/4.0 Opus** | 頂級推理（邏輯與規劃） | 慢 | 高 (約 Sonnet 2.5倍) | **技術藍圖設計**。適合前期架構規劃、高難度數學/物理推導，不建議用於日常瑣碎問答。 |
| **Claude 3.5 Haiku** | 中等（日常對話與快速處理） | 極快 | 低 (約 Sonnet 1/5) | **低成本極速首選**。適合代碼語法糾錯、大批次文字翻譯、簡單分類與過濾任務。 |

## 4. CLAUDE.md 優化規則：將錯誤率從 41% 降至 3%

在 [[Claude 專案管理一桌三櫃工作流|一桌三櫃工作流]] 中，`CLAUDE.md` 是引導 Claude Code 的本地專案指導書。然而，若規則寫得越加越亂，反而會導致 AI 執行時發生認知偏差，錯誤率飆升。

### 12 條優化核心規則（精華）
1.  **指令具體化**：只寫入專案的**建置指令**（如 `npm run dev`）與**測試指令**（如 `pytest`），不可包含模糊的指導性 Prompt。
2.  **單一事實來源**：一條開發規則只在一個地方聲明，避免在 `CLAUDE.md` 和 `.cursorrules` 中重複出現，防止規則飄移。
3.  **定期重構與精簡**：保持文件在 100 行以內。

### 🛠️ 規則自我審查 Prompt 範本
您可以定期將 `CLAUDE.md` 丟給 Claude，並使用以下 Prompt 進行規則瘦身：

```markdown
請作為高級軟體架構師，對本專案的 CLAUDE.md 文件進行自我審查。請依據以下 5 個條件，揪出多餘、矛盾或低效的設定，並輸出一個重構精簡後的版本：
1. 是否含有重複或語意相似的開發規則？
2. 是否有與當前代碼結構不符的過時指令？
3. 是否存在模糊不清、容易讓 AI 產生歧義的描述？
4. 規則是否過於冗長，能用更緊湊的技術術語概括？
5. 是否符合「只寫建置/測試指令與核心規範，不寫對話指導」的輕量原則？
```
經過此自我審查重構，能讓 Claude Code 遵循規則的錯誤率顯著降低。
